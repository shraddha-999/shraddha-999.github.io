# Credit risk scoring engine

class CibilScoreCalculator
  TARGET_MAX     = 15
  MIN_TO_PROCEED = 10

  SECTION_WEIGHTS = {
    worst_dpd:           1.67,  # heaviest — DPD history
    enquiries:           1.17,
    micro_loans:         0.83,
    cibil_score:         0.67,
    active_loans:        0.50,
    settlement_writeoff: 0.17
  }.freeze

  module HardDeclineReason
    ACTIVE_NPA = "Active NPA detected"
    WRITE_OFF  = "Write-off / Settlement detected"
  end

  def calculate
    weighted = @sections.sum { |s| s[:weighted_score] }
    {
      proceed:      !@hard_decline && weighted >= MIN_TO_PROCEED,
      hard_decline:  @hard_decline,
      total_score:   weighted
    }
  end
end

RateGuard

Distributed Rate Limiter + SaaS Dashboard

Production-grade HTTP rate limiter implementing three algorithms with atomic Redis Lua scripts — no race conditions under any concurrency. Paired with a full React 19 dashboard: real-time charts, animated algorithm visualizer, request simulator, and Prometheus metrics parsing.

Core problem

Under concurrent load, check-then-write races let multiple requests through above the limit. Without atomicity, every distributed rate limiter is eventually wrong.

Solution

All three algorithms execute as atomic Lua scripts via Redis EVALSHA — the entire read-modify-write is a single Redis operation. 49 tests pass including thread-safety stress tests with zero race conditions observed.

Python · FastAPI Redis · Lua React 19 · TypeScript Prometheus · Grafana Docker Compose
